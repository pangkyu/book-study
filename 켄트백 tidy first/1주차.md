# 1주차

## 1장 보호 구문

중첩된 if문은 적게 쓰는 것이 읽기 좋습니다.

```js
if(조건 부정) return
if(다른 조건) return

```

코드 세부 사항을 살펴보기 전에 염두에 두어야 할 몇 가지 전제 조건이 있다 라고 알 수 있습니다.

보호 구문이 있는 코드라면 전제 조건이 명시적으로 드러날 때 분석하기 쉽습니다.

단, 남용하는 경우 읽기가 매우 까다롭습니다.

## 2장 안 쓰는 코드

실행되지 않는 코드는 지웁니다.
혹시 나중에 필요하다고 생각이 드는 경우 형상 관리 도구로 복구하면 됩니다.

코드 리팩토링할때는 조건 내용을 택해 true로 줄일 수 있고, 루틴 하나, 파일 하나, 디렉토리 하나 정도로 확장할 수 있습니다

## 3장 대칭으로 맞추기

코드가 유기체로 성장하면 점점 코드가 읽기 어려워질 수 있습니다.
한가지 패턴의 코드는 읽다 보면 익숙해지나 두 가지 이상의 패턴을 섞어 쓰면 혼란스럽습니다.

```js
foo(){
    if문으로 코드를 쓰는 경우
}

foo() {
    return 삼항 ? 연산 : 자로 쓰는 경우
}
```

하나의 방식으로 통일하여 코드를 작성해 로직 파악을 쉽게 합니다.

## 4장 새로운 인터페이스로 기존 루틴 부르기

루틴을 호출할 때 기존 인터페이스 때문에 어렵거나 복잡한 경우 새로운 인터페이스를 새롭게 구현합니다.

새롭게 구현한 통로 인터페이스는 소프트웨어 설계에서 작은 단위로 중추적 역할을 합니다.

어떤 동작을 변경해야 할 때가 왔을 때, 통로 인터페이스를 이용해 설계했다면 변경하기가 한층 더 수월합니다.

아래처럼 설계하는 경우를 추천합니다.

거꾸로 코딩하기 : 루틴의 마지막 줄부터 시작합니다. 이때, 마지막 줄에 이르기까지 필요한 결과를 모두 확보한 듯이 짭니다.
테스트 우선 코딩 : 테스트부터 작성하여 통과 요건을 정합니다.
도우미 설계 : 만일 나에게 특정 업무를 해주는 루틴,객체,서비스가 있다면 나머지 작업은 쉽습니다.

## 6장 응집도를 높이는 배치

코드의 순서를 바꿔서 변경할 요소를 가까이 두면 됩니다. 같은 디렉토리에 들어있는 파일에 적용할 수 있습니다.

결합도를 제거하는 것이 가장 좋은 코드 정리지만 리소스 문제로 정리하지 못할 수 있습니다

응집도를 높이는 순서로 정리하면 코드를 더 쉽게 변경할 수 있습니다.

## 7장 선언과 초기화를 함께 옮기기

변수 선언과 초기화 위치는 종종 떨어져 있기도 합니다.
변수 초기화는 이름이 주는 의미를 더 강화합니다. 변수 초기화는 이름이 주는 의미를 더 강화합니다.

```js
fn(){
    int a
    ...code
    a = ...
    int b
    ... a를 사용할 수 있으나 b를 사용하지 않는 코드
    b = ...a...
    ... b를 사용하는 코드
}
```

초기화를 변수 선언 근처로 이동하여 문제를 해결합니다.

```js
fn(){
    int a = ...
    ... a를 사용하지 않는 코드
    ... a를 사용하고 bsms 사용하지 않는 코드

    int b = ...a...
    ... b를 사용하는 코드
}
```

## 8장 설명하는 변수

코드는 처음에는 작아도 시간이 흐르면 점점 커집니다.
어렵게 크고 복잡한 코드 표현식을 이해했다면 전체에서 일부 표현식을 추출한 후, 표현식의 의도가 드러나도록 변수 이름을 만들어 할당합니다.

```js
return new Point(
    ...긴 표현식...,
    ...다른 긴 표현식...
)
```

```js
x = ...긴 표현식...
y = ...다른 긴 표현식...

return new Point(x,y)
```

표현식을 분리하면 다음 번 코드를 변경할 때 둘 중 하나만 읽으면 되니까 재빨리 변경할 수 있습니다.

## 10장 명시적인 매개변수

변경하려는 코드를 읽는 중에 루틴에서 다루고 있는 일부 데이터가 명시적으로 전달되지 않는 경우, 입력을 명확하게 하려면 어떻게 해야 할까요

```js
params = { a : 1, b : 2}
foo(params)

function foo(params){
    ...params.a...,...params.b..
}
```

foo를 나누면 명시적 매개변수 정리를 적용할 수 있습니다.

```js
function foo(parmas){
    foo_body(params.a, params.b)
}

function foo_body(a,b){
    ...a... , ...b...
}
```

매개변수를 명시적으로 드러나게 만든 다음 함수를 연쇄적으로 호출할 수 있게 준비합니다.

이렇게 만들면 코드 읽기와 테스트, 분석이 쉬워집니다.

## 11장 비슷한 코드끼리

## 12장 도우미 추출(헬퍼)

코드를 보다 루틴 속 코드 중에서, 목적이 분명하고 나머지 코드와는 상호작용이 적은 코드 블록을 만날 때가 있습니다.

헬퍼로 추출한 후 이름을 붙이는데, 이때 목적에 따라 네이밍을 짓습니다.

## 13장 하나의 더미

필요한 만큼의 코드를 하나의 더미처럼 느껴질 때까지 흩어진 코드르를 모으고 정리합니다.

코르 정리를 선행하면 더 작은 조각 단위로 결합을 제거하는 길을 제시하여 응집도를 높일 수 있습니다.

작은 코드 조각을 지향하는 목적은 코드를 한 번에 조금씩 이해할 수 있도록 하는 것입니다. 단, 작은 코드 조각들이 서로 교류하는 방식이 더 어려워질 수 있기 때문에 코드를 한데 모아 이해하기 어려운 부분을 추출해서 새롭게 정리해야 합니다.

## 14장 설명하는 주석

파일 앞에 주석이 없는 경우, 설명을 추가하여 파일을 읽을 사람이 얻게 될 유용함을 미리 알려줍니다.

코드의 결함을 발견했다면 해당 위치에 주석을 달아 표시합니다.

## 15장 불필요한 주석 지우기

코드만으로 내용을 모두 이해할 수 있으면 주석은 삭제합니다.
