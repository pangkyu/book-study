## 3주차

### 전체 조감도 파악

클로드가 디렉토리 구조를 탐색하고 리드미, 패키지를 분석하여 개요를 정리합니다.

세부 코드부터보면 길을 잃기 때문에, 반드시 전체 구조부터 시작합니다.

코드 베이스가 10만줄이 넘는 경우에는 `/clear`로 세션을 정리하면서 영역별로 나누어 탐색합니다.

### 정확한 지시

모호한 지시보다는 정확하게 지시한 경우 시간도 단축되고 조금 더 정확하게 동작합니다.

```md
@auth/login.py의 authenticate 함수에 이메일 형식 검증을 추가해줘.
정규식으로 @와 도메인이 포함되었는지 확인하고,
유효하지 않으면 ValueError("Invalid email format")를 발생시켜줘.
```

한 번에 2~3개 파일이 적당하고 5개 이상인 경우 컨텍스트가 넘칠 수 있습니다

### TDD

테스트를 먼저 작성한 뒤 코드를 작성해달라고 단계를 나누면 red-green-refactor가 정확히 동작합니다.

**Red : 테스트 먼저 작성**

```md
이메일 유효성 검사 함수 validateEmail(email)을 만들 겁니다.

요구사항:

- 유효한 이메일이면 { valid: true }를 반환
- 유효하지 않으면 { valid: false, reason: "에러 메시지" }를 반환
- 검사 항목: @ 포함, @ 앞뒤 문자 존재, 도메인에 점 포함, 공백 불가, null/빈 문자열 불가

테스트를 먼저 작성해줘. 구현은 아직 하지 마.
파일: src/validateEmail.test.js
```

**Green : 최소한의 구현**

```md
@src/validateEmail.test.js의 모든 테스트를 통과하는 최소한의 구현을 작성해줘.
파일: src/validateEmail.js
과도한 최적화나 추가 기능은 넣지 마.
```

이후 엣지 케이스를 찾도록 시킵니다

```md
@src/validateEmail.test.js에 엣지 케이스 테스트를 추가해줘:

- 특수문자 포함 이메일 (user+tag@domain.com)
- 연속된 점 (user@domain..com)
- 매우 긴 이메일 주소

추가한 테스트가 실패하면 @src/validateEmail.js 구현도 함께 수정해줘.
```

실패하는 테스트가 있으면 구현을 또 수정합니다

**주의 사항**

- red 단계를 건너뛰지 않습니다. 테스트가 실패하는 것을 반드시 확인한 다음 구현을 시작합니다
- 한 사이클에 하나의 기능만 다룹니다
- 테스트와 구현을 동시에 요청하지 않습니다

### 리팩토링

코드 정리를 해달라고 바로 명령하는 것이 아닌 계획을 먼저 세우고 파일 단위로 진행하도록 합니다.

계획이 의도와 다르면 방향을 수정한 뒤, 실행합니다

### hooks 자동 품질검사

훅은 클로드 동작 시점에 자동으로 실행되는 명령어입니다. 4가지 타입을 지원합니다

- command : 셸 명령어 실행
- prompt : 별도의 클로드 모델에게 프롬포트를 보내 판단을 받습니다
- agent : 파일 읽기, 코드 검색 등 도구를 사용할 수 있는 서브 에이전트를 호출하여 다단계 검증 수행
- http : 지정한 URL로 json을 post 전송(외부 서비스 연동용)

**사용 방법**

```md
Claude가 작업을 마치고 내 입력을 기다릴 때 macOS 데스크탑 알림을 보내주는 Hook을 설정해줘.
```

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"작업이 완료되었습니다\" with title \"Claude Code\" sound name \"Glass\"'"
          }
        ]
      }
    ]
  }
}
```

클로드가 실행하는 bash 명령어 중 위험한 패턴을 사전에 차단할 수 있습니다. ex) drop table, rm -rf 등

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/block-dangerous-commands.sh",
            "timeout": 5,
            "statusMessage": "Checking for dangerous commands..."
          }
        ]
      }
    ]
  }
}
```

### github actions

1. api키를 github secrets에 등록합니다

api 키는 클로드 구독과 별도의 종량제 과금입니다.

```md
name: Claude Code Review

on:
pull_request:
types: [opened, synchronize]
issue_comment:
types: [created]

jobs:
review:
if: |
(github.event_name == 'pull_request') ||
(github.event_name == 'issue_comment' && contains(github.event.comment.body, '@claude'))
runs-on: ubuntu-latest
permissions:
contents: read
pull-requests: write
issues: write
steps: - name: Checkout repository
uses: actions/checkout@v5
with:
fetch-depth: 1

      - name: Run Claude Code
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
```

- 해당 파일을 적용한 뒤, 코멘트에 `@claude`를 포함하면 클로드가 해당 요청에 대해 분석하고 답변합니다

### github worktree 병렬세션

기능 개발 중 브랜치를 전환하면 실행 중이던 클로드 세션이 종료됩니다.

`claude -w name`으로 독립 세션을 띄우면 기존 작업을 중단하지 않고 병렬로 처리할 수 있습니다

`git checkout`이 아닌 `git worktree`를 사용하면 브랜치별로 독립된 디렉토리가 생기고 세션이 끊기지 않습니다

```md
# 기능 개발용 세션

claude -w feature-login

# 버그 수정용 세션 (다른 터미널에서)

claude -w fix-auth-error

# 리팩토링용 세션 (다른 터미널에서)

claude -w refactor-api
```

`/loop`으로 주기적으로 작업 모니터링을 할 수 있습니다.

각 워크트리 세션에서 작업이 끝나면 클로드에게 커밋과 pr 생성을 요청합니다.
pr기반 워크플로우의 변경사항을 독립적으로 리뷰하고 머지할 수 있습니다.

`-w`플래그로 생성한 워크트리는 세션 종료 시 자동 정리 여부를 묻습니다.

클로드가 `git worktree list`로 목록을 확인하고 머지된 워크트리를 `git worktree remove`로 삭제합니다

### skills

skills는 클로드 기능을 확장하는 재사용 가능한 지시사항입니다.

직접 개발하거나 내장된 스킬을 사용할 수 있고, 커뮤니티에 있는 skills를 활용하는 방법도 있습니다.

`/plugin marketplace add anthropics/skills`
`/plugin marketplace add <github-owner>/<repo>`

`/skills`로 설치된 스킬 목록을 확인합니다.

`.claude/skills/` : 현재 프로젝트. 프로젝트 별 스킬, 깃 커밋으로 팀 공유
`~/.claude/skills/` : 모든 프로젝트. 개인 전역 스킬

같은 이름의 스킬이 여러 위치에 있으면 프로젝트 스킬이 우선됩니다

### 스킬 직접 개발하기

배포, 메시지 전송 등 부작용이 있는 스킬은 반드시 `disable-model-invocation : true`를 설정합니다
`context : fork`를 설정한 스킬은 서브 에이전트로 실행되어 메인 컨텍스트에 영향을 주지 않습니다.
`!command` 동적 컨테스트는 스킬 로드 시 즉시 실행됩니다
스킬 디렉토리명이 명령어 이름입니다. `review/skill.md`는 `/review`로 호출됩니다
내장 명령어와 이름이 겹치지 않도록 주의합니다.
